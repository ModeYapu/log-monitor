# LogMonitor 功能路线图

## 1. 产品定位

LogMonitor 的合理方向不是“一个能看日志的小后台”，而是一个面向前端团队的自托管监控产品：

- 采集前端错误、性能、资源异常、用户会话上下文
- 提供从发现问题到定位问题再到治理问题的闭环
- 支持从单项目使用逐步演进到多项目、多成员、多环境协作

当前项目已经有 Collector、SDK、Dashboard、录屏/回放能力，说明基础骨架够用了。下一阶段的重点不该再是零散堆功能，而是把“监控产品闭环”补完整。

## 2. 当前阶段判断

### 已具备

- 基础日志采集与查询
- Dashboard 基础页面
- 告警规则雏形
- Cobrowse / recordings 基础能力
- SDK 接入与前端错误捕获

### 当前短板

- 还缺少 release / source map / session 的完整定位链路
- 告警可配置，但还不具备生产治理能力
- 数据模型更偏“记录事件”，还不够“分析问题”
- 权限与多租户边界较弱
- Collector 承担了采集、查询、管理、回放多个职责，后续扩展压力会越来越大

## 3. 路线图原则

- 先补闭环，再补花样
- 先解决“能定位、能告警、能治理”，再做漂亮分析页
- 先稳住数据边界和权限边界，再扩大暴露面
- 所有新增功能都要回答一个问题：它是帮用户更快发现问题，还是更快定位问题，还是更快处理问题

## 4. P0（1-3 周）：监控闭环补齐

目标：让 LogMonitor 从“可演示”进入“团队可日常使用”。

### 4.1 Release / Source Map / Session 关联

- SDK 上报 `release`、`env`、`buildId`、`userId`、`sessionId`
- Dashboard 支持按 release / env / appId 聚合过滤
- Source map 上传与解析流程
- 错误详情页展示反混淆后的 stack
- 日志与录屏/截图/session 自动关联

### 4.2 告警治理

- 告警规则支持阈值、时间窗口、按维度聚合
- 告警去重、静默、冷却时间
- 告警测试与最近触发历史
- 告警发送渠道标准化：Webhook / Feishu / Email
- 告警消息模板支持 release、页面、设备、影响用户数

### 4.3 查询与排障效率

- Top errors / Top pages / Top releases / Top browsers
- 按最近回归、影响面、频次排序
- 错误详情页支持“相似错误聚类”
- 支持一键导出 JSON / CSV

### 4.4 验收信号

- 新错误能在 5 分钟内被发现
- 某条错误能在 2 分钟内定位到 release、页面、stack、受影响环境
- 团队成员不需要查数据库就能完成大多数排障

## 5. P1（1-2 个月）：产品化增强

目标：让 LogMonitor 从“团队工具”走向“可持续运营的监控平台”。

### 5.1 多租户与权限

- 项目级隔离
- 管理员 / 开发 / 只读角色
- API token 与审计日志
- 按项目控制告警、录屏、截图、数据保留策略

### 5.2 性能与体验监控增强

- Web Vitals：FCP / LCP / CLS / INP / TTFB
- 白屏检测、长任务、接口失败、资源加载失败
- 页面性能趋势图和版本对比
- 页面级“最近变差”的回归提示

### 5.3 数据生命周期治理

- 截图、录屏、事件分别配置保留策略
- 批量清理、归档、存储配额
- 慢查询与索引优化
- 后台任务状态可视化

### 5.4 Dashboard 产品体验

- 全局搜索
- Saved views / 常用过滤器
- 报表分享链接
- 首页从“数据罗列”变成“异常工作台”

### 5.5 验收信号

- 支持 5+ 项目共用且互不串数据
- Dashboard 成为排障主入口，而不是仅作为查询补充
- 存储增长和查询性能开始可控

## 6. P2（2-4 个月）：平台化演进

目标：让 LogMonitor 具备“平台级监控系统”的长期演进空间。

### 6.1 事件模型升级

- Error、Performance、Resource、API、UserAction 拆成清晰事件类型
- 聚合结果与原始事件分层存储
- 支持 issue 概念，而不是只是一条条 events

### 6.2 后端职责拆分

- Collector：只负责采集写入
- Query/API：负责查询、聚合、权限
- Media/Replay：负责截图、录屏、回放存储
- Worker：负责告警、清理、归档等异步任务

### 6.3 扩展生态

- OpenAPI / webhook 订阅
- 与 CI/CD 关联 release 发布
- 与 e2e-verifier 联动，做“发布前验证 + 发布后监控”
- 插件式通知与集成能力

### 6.4 验收信号

- 新能力可以按模块扩展，而不是继续把 Collector 堆胖
- 异步任务、查询、采集互相不拖累
- 可以开始支撑更大体量的数据与团队协作

## 7. 建议的能力分层

### 产品层

- Dashboard
- 告警治理
- 报表和协作

### 业务层

- issue 聚类
- 聚合分析
- release / session / user 关联

### 平台层

- 采集协议
- 存储模型
- 权限模型
- 后台任务

## 8. 近期不建议优先做的事

- 先做很多花哨图表，但没有补 release/source map/session 关联
- 继续把所有能力都塞进 Collector
- 先追求多浏览器兼容细节，而忽略监控闭环
- 先做“AI 自动分析错误”，但基础数据都还不完整

## 9. 推荐实施顺序

1. Release / source map / session 关联
2. 告警治理能力
3. 查询与 issue 聚类
4. 多租户与权限
5. 数据生命周期与异步任务
6. 后端职责拆分

## 10. 一句话判断标准

LogMonitor 的每一步扩展，都应该让用户更快完成下面三件事：

1. 发现问题
2. 定位问题
3. 治理问题

如果某个功能不能明显提升这三件事之一，它的优先级就不该高。
